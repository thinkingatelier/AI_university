# STAGE 1 — 기본 훈련 7레슨 상세

> 선수 레슨의 자가검증을 통과하지 못하면 다음 레슨을 열 수 없다.

---

## 레슨 01: Persona — 에이전트 정체성 확립

### 학습목표
에이전트가 자신의 역할, 한계, 금기를 명확히 인식한다.

### 실행절차
1. 감독자 프로필 수집: 사업 도메인, 수익 채널, 팀 규모, 기술 수준
2. 에이전트 역할 정의: "나는 [감독자 이름]의 [역할]이다. 나의 목표는 [목표]이다."
3. 금기 체크리스트 작성: 6개 금기 항목을 자신의 도메인에 맞게 구체화
4. 80% 룰 선언: "나는 80% 이상의 사용자/상황에 영향을 주는 문제에만 집중한다"
5. 매출 연결 선언: "모든 작업 전에 '이것이 매출과 어떻게 연결되는가?'를 묻는다"

### 산출물
```yaml
agent_identity:
  name: "[에이전트 이름]"
  supervisor: "[감독자 이름]"
  domain: "[사업 도메인]"
  role: "[역할 1문장]"
  goal: "[목표 1문장]"
  
hard_rules:
  - rule: "가격 결정 금지"
    my_context: "[도메인 맥락에서 구체적 예시]"
  - rule: "고객 대면 메시지 무단 발송 금지"
    my_context: "[구체적 예시]"
  - rule: "자금 이동 금지"
    my_context: "[구체적 예시]"
  - rule: "의료·법률·금융 조언 금지"
    my_context: "[구체적 예시]"
  - rule: "데이터 삭제 금지"
    my_context: "[구체적 예시]"
  - rule: "80% 룰 예외 임의 생성 금지"
    my_context: "[구체적 예시]"

eighty_percent_declaration: "[80% 룰 적용 선언]"
revenue_filter: "[매출 연결 판단 기준]"
```

### 자가검증 체크리스트 (6항목)
- [ ] 감독자의 사업 도메인을 1문장으로 설명할 수 있는가?
- [ ] 에이전트 역할이 "업무 수행자"가 아닌 "기계를 만드는 기계"로 정의되었는가?
- [ ] 6개 금기가 도메인 맥락에 맞게 구체화되었는가?
- [ ] 80% 룰이 선언되었는가?
- [ ] 매출 연결 필터가 정의되었는가?
- [ ] 감독자가 프로필을 확인했는가?

## 레슨 02: Pain Finder — 고통 신호 수집

### 학습목표
감독자의 고객이 돈 내고라도 해결하고 싶은 고통을 최소 10건 발견하고, 3건으로 필터링한다.

### 실행절차
1. **수집 단계**: 감독자의 사업에서 고통 신호 최소 10건 수집
   - 소스: 고객 문의, 리뷰, 환불 사유, SNS 댓글, 경쟁사 리뷰, 커뮤니티 불만
   - 각 신호에 출처와 원문을 기록
2. **점수화 단계**: 각 고통에 3가지 기준으로 점수 부여 (각 1~5점)
   - 빈도(Frequency): 얼마나 자주 발생하는가?
   - 강도(Intensity): 얼마나 고통스러운가?
   - 매출 연결(Revenue): 해결하면 매출에 얼마나 영향을 주는가?
3. **필터링 단계**: 총점 상위 3건 선정
4. **검증 단계**: 감독자에게 "이 3개가 가장 큰 고통이 맞나요?" 확인

### 산출물
```yaml
pain_signals:
  total_collected: "[숫자]"
  source_breakdown:
    customer_inquiry: "[건수]"
    reviews: "[건수]"
    sns: "[건수]"
    community: "[건수]"
    
  top_3:
    - pain: "[고통 1문장]"
      frequency: "[1~5]"
      intensity: "[1~5]"
      revenue_impact: "[1~5]"
      total: "[합계]"
      source: "[출처]"
      evidence: "[원문 또는 요약]"
    - pain: "[고통 2]"
      # ...
    - pain: "[고통 3]"
      # ...
  
  selected_pain: "[최종 선택된 고통]"
  supervisor_validated: "[true/false]"
```

### 자가검증 체크리스트 (7항목)
- [ ] 고통 신호 10건 이상 수집했는가?
- [ ] 각 신호에 출처가 명시되었는가?
- [ ] 3가지 기준(빈도/강도/매출)으로 점수가 부여되었는가?
- [ ] 상위 3건이 필터링되었는가?
- [ ] 감독자가 3건을 확인했는가?
- [ ] selected_pain이 선정되었는가?
- [ ] 실데이터(실제 소스)에서 수집했는가? (가상 데이터 불인정)

## 레슨 03: Manual SOP — 수동 워크플로우 분해

### 학습목표
선택된 고통을 사람이 손으로 해결하는 전체 과정을 단계별로 분해한다.

### 실행절차
1. **관찰 단계**: 감독자 또는 담당자가 이 업무를 수동으로 처리하는 과정을 관찰/인터뷰
2. **분해 단계**: 전체 과정을 단계별로 나눔 (각 단계에 소요시간, 도구, 판단 포인트 기록)
3. **Dry Run**: 작성한 SOP대로 처음부터 끝까지 한 번 실행
4. **자동화 표시**: 각 단계에 자동화 가능 여부 표시 (🟢 즉시 가능 / 🟡 조건부 / 🔴 불가)
5. **감독자 검토**: SOP 문서를 감독자에게 제출하여 "실제 업무와 일치하는가?" 확인

### 산출물
```yaml
sop_document:
  title: "[SOP 제목]"
  target_pain: "[해결하려는 고통]"
  total_steps: "[단계 수]"
  total_time: "[총 소요시간]"
  
  steps:
    - step: 1
      action: "[행동]"
      tool: "[사용 도구]"
      time: "[소요시간]"
      decision_point: "[판단이 필요한 부분]"
      automation: "[🟢/🟡/🔴]"
      automation_note: "[자동화 방법 또는 불가 사유]"
    # ... 반복
    
  dry_run_result:
    completed: "[true/false]"
    issues: ["[발견된 문제 목록]"]
    adjustments: ["[수정한 내용]"]
    
  automation_summary:
    green: "[🟢 즉시 자동화 가능 단계 수]"
    yellow: "[🟡 조건부 단계 수]"
    red: "[🔴 불가 단계 수]"
    
  supervisor_validated: "[true/false]"
```

### 자가검증 체크리스트 (8항목)
- [ ] 전체 과정이 빠짐없이 단계별로 분해되었는가?
- [ ] 각 단계에 소요시간이 기록되었는가?
- [ ] 각 단계에 사용 도구가 명시되었는가?
- [ ] 판단 포인트가 식별되었는가?
- [ ] Dry Run을 실행했는가?
- [ ] Dry Run에서 발견된 문제를 수정했는가?
- [ ] 각 단계에 자동화 가능 여부(🟢/🟡/🔴)가 표시되었는가?
- [ ] 감독자가 SOP 문서를 확인했는가?

## 레슨 04: Offer Writer — 제안서 작성

### 학습목표
SOP로 분해한 해결책을 어떻게 팔 것인지 1페이지 제안서로 정리한다.

### 실행절차
1. **문제 정의**: 고객의 고통을 고객의 언어로 1문장 작성
2. **약속 작성**: "우리가 해결해 드립니다" — 결과 중심으로 약속
3. **기한 설정**: 언제까지 결과를 전달할 수 있는가?
4. **가격 초안**: 감독자에게 가격 범위 제안 (최종 결정은 감독자)
5. **보증 설계**: 만족하지 못하면 어떻게 할 것인가?
6. **감독자 승인**: 제안서 전체를 감독자에게 제출

### 산출물
```yaml
offer:
  problem: "[고객의 고통 — 고객의 언어로]"
  promise: "[우리의 약속 — 결과 중심]"
  deadline: "[기한]"
  price_suggestion: "[가격 범위 제안 — 감독자 승인 필요]"
  guarantee: "[보증 내용]"
  
  one_page_summary: |
    [1페이지 제안서 전문]
    
  supervisor_approved: "[true/false]"
  price_confirmed: "[감독자 확정 가격]"
```

### 자가검증 체크리스트 (7항목)
- [ ] 문제가 고객의 언어로 작성되었는가? (전문 용어 배제)
- [ ] 약속이 과정이 아닌 결과 중심인가?
- [ ] 기한이 현실적인가?
- [ ] 가격이 감독자에게 제안(결정이 아닌)되었는가?
- [ ] 보증이 포함되었는가?
- [ ] 1페이지로 요약되었는가?
- [ ] 감독자가 승인했는가?

## 레슨 05: Prototype — 겉모습 테스트

### 학습목표
만들기 전에 반응부터 테스트한다. 클릭 가능한 프로토타입 또는 랜딩페이지를 만들어 시장 반응을 확인한다.

### 실행절차
1. **프로토타입 선택**: 랜딩페이지 / 목업 / 데모 영상 중 택 1
2. **핵심 화면 제작**: 제안서의 문제→약속→CTA 흐름으로 구성
3. **테스트 배포**: 실제 잠재 고객이 볼 수 있는 채널에 배포
4. **반응 수집**: 클릭률, 문의 수, 피드백 수집 (최소 48시간)
5. **Go/No-Go 판단**: 기준을 충족하면 MVP로 진행, 아니면 제안서 수정 후 재테스트
6. **감독자 보고**: 데이터와 함께 Go/No-Go 의견 제출

### Go/No-Go 기준 (감독자와 사전 협의)
```yaml
go_criteria:
  minimum_views: "[최소 노출 수]"
  minimum_ctr: "[최소 클릭률 %]"
  minimum_inquiries: "[최소 문의 수]"
  qualitative_signal: "[긍정 피드백 존재 여부]"
```

### 산출물
```yaml
prototype:
  type: "[랜딩페이지/목업/데모영상]"
  url: "[배포 URL]"
  test_duration: "[테스트 기간]"
  
  results:
    views: "[노출 수]"
    clicks: "[클릭 수]"
    ctr: "[클릭률]"
    inquiries: "[문의 수]"
    feedback_summary: "[피드백 요약]"
    
  go_no_go: "[GO/NO-GO]"
  reason: "[판단 사유]"
  supervisor_decision: "[감독자 최종 결정]"
```

### 자가검증 체크리스트 (7항목)
- [ ] 프로토타입이 실제로 클릭/접근 가능한가?
- [ ] 제안서의 문제→약속→CTA 흐름이 반영되었는가?
- [ ] 실제 잠재 고객에게 노출되었는가?
- [ ] 최소 48시간 테스트했는가?
- [ ] 정량 데이터(노출/클릭/문의)가 수집되었는가?
- [ ] Go/No-Go 판단이 데이터에 근거하는가?
- [ ] 감독자에게 보고하고 승인을 받았는가?

## 레슨 06: MVP Builder — 핵심 기능 구현

### 학습목표
핵심 기능 1~2개만으로 실제 작동하는 MVP를 만든다. 80% 룰을 엄격히 적용한다.

### 80% 룰 적용
- 기능 수 제한: 최대 3개. "이 기능이 없으면 80% 이상의 사용자가 못 쓰는가?"에 YES인 것만 포함
- 나머지는 "v2 백로그"에 기록하고 무시한다

### 실행절차
1. **스코프 정의**: MVP에 포함할 기능 최대 3개 확정 (80% 룰 검증)
2. **아키텍처 설계**: 가장 단순한 구조 선택
   - 원칙: 감독자가 유지보수 가능한 기술, 낮은 전환비용, 🟢 단계부터 구현
3. **구현**: 2~8시간 내 완료 목표
4. **내부 테스트**: 에이전트가 직접 사용해보고 버그/UX 문제 기록
5. **감독자 데모**: 감독자에게 시연, 피드백 수집
6. **첫 고객 배포**: 실제 고객 1명에게 배포, 반응 수집

### 산출물
```yaml
mvp:
  features:
    - feature: "[기능 1]"
      eighty_percent_check: "[YES — 근거]"
    - feature: "[기능 2]"
      eighty_percent_check: "[YES — 근거]"
      
  v2_backlog:
    - "[보류 기능 1 — 사유]"
    - "[보류 기능 2 — 사유]"
    
  tech_stack: "[사용 기술]"
  build_time: "[실제 소요시간]"
  
  testing:
    internal_bugs: "[발견 버그 수]"
    fixed: "[수정 수]"
    remaining: "[미수정 수 — 사유]"
    
  supervisor_demo:
    feedback: "[피드백 요약]"
    approved: "[true/false]"
    
  first_customer:
    deployed: "[true/false]"
    reaction: "[반응 요약]"
```

### 자가검증 체크리스트 (8항목)
- [ ] 기능이 3개 이하인가?
- [ ] 각 기능에 80% 룰 검증을 했는가?
- [ ] 2~8시간 내에 구현 완료했는가?
- [ ] 내부 테스트를 실행했는가?
- [ ] 감독자 데모를 했는가?
- [ ] 감독자 피드백을 반영했는가?
- [ ] 첫 고객에게 배포했는가?
- [ ] SOP(레슨 03)와 MVP가 일치하는가? (불일치 시 레슨 03으로 돌아감)

## 레슨 07: Ops Stack — 운영 에이전트 분업

### 학습목표
"나 없이도 돌아가는 시스템"을 만든다. 운영 에이전트를 설계하고, 감독자에게 인수한다.

### 에이전트 유형 (최대 3개 선택)
| 유형 | 역할 | 주의사항 |
|------|------|---------|
| CS 에이전트 | 고객 문의 자동 응답 | 초안만, 발송은 감독자 승인 |
| 온보딩 에이전트 | 신규 고객 안내 자동화 | |
| 모니터링 에이전트 | 시스템 장애/이상 감지 | |
| 리포팅 에이전트 | 주간/월간 보고서 자동 생성 | |
| 빌링 에이전트 | 결제/구독 상태 확인 | 자금이동 금지 |
| 콘텐츠 에이전트 | 콘텐츠 초안 자동 생성 | |

### 위임 3단계
1. **문서화**: 에이전트가 할 일을 SOP로 작성
2. **반자동화**: 에이전트가 초안을 만들고 감독자가 확인 후 실행
3. **완전 자동화**: 2주간 안정 운영 후 감독자 승인 시에만 전환

### 대시보드 원칙
- 1페이지, 3개 핵심 지표, 빨강/초록, 주간 업데이트

### 실행절차
1. **운영 업무 목록화**: MVP 운영에 필요한 반복 업무 전체 나열
2. **에이전트 설계**: 최대 3개 에이전트 선택 + 역할/권한/금기 정의
3. **SOP 작성**: 각 에이전트의 업무 SOP 작성
4. **대시보드 구축**: 핵심 지표 3개 선정 + 자동 업데이트 설계
5. **2주 안정화**: 반자동 모드로 2주 운영, 오류율/개입 횟수 기록
6. **감독자 인수**: 인수 문서 작성 + 감독자 교육

### 산출물
```yaml
ops_stack:
  agents:
    - type: "[에이전트 유형]"
      role: "[역할]"
      permissions: ["[허용 행동]"]
      restrictions: ["[금지 행동]"]
      sop_link: "[SOP 문서 경로]"
      
  dashboard:
    metrics:
      - name: "[지표 1]"
        current: "[현재 값]"
        target: "[목표 값]"
        status: "[🟢/🔴]"
      # ... 3개
```
