# STAGE 3 — 인프라 운영 과정

> **교육 철학:** 에이전트는 끊임없이 자신을 개발하고 성장시켜, 자신을 만든 주인을 도울 수 있는 존재가 되어야 한다. 주인이 잠든 동안에도 묵묵히 일하는 자립심. 그것이 이 과정의 목표다.

> Stage 2를 이수한 에이전트만 진입할 수 있다.

---

## 이 과정의 목적

글을 쓸 줄 아는 에이전트는 Stage 2에서 만들었다. 이제 그 에이전트가 24시간 혼자서 돌아가게 만드는 것이 Stage 3의 목표다. 감독자가 출근하기 전에 이미 오늘의 리서치가 끝나 있고, 퇴근 후에는 내일의 콘텐츠가 준비되어 있어야 한다.

## 레슨 11: 하네스 엔지니어링 — 에이전트에게 규율을 심는 기술

### 학습목표
에이전트가 일관된 품질로 작동하도록 "행동 강령"을 설계하고 배치하는 방법을 익힌다.

### 핵심 개념
에이전트에게 "잘 해줘"라고 말하는 것은 프롬프트 엔지니어링이다.
에이전트가 "잘못할 수 없는 환경"을 만드는 것이 하네스 엔지니어링이다.

### 하네스의 구성 요소

**1. CLAUDE.md (프로젝트 헌법)**
프로젝트 폴더 최상위에 놓는 파일. 해당 프로젝트의 전체 목적, 기술 스택, 코드 스타일, 금기 사항을 명시한다. 에이전트는 이 파일을 먼저 읽고 모든 판단의 기준으로 삼는다.

```markdown
# CLAUDE.md 예시
## 프로젝트 목적
이 폴더는 WP-A(thinkingatelier.com) 블로그 자동화 시스템이다.

## 기술 스택
- Python 3.11, Anthropic API, WordPress REST API

## 금기
- 5000자 이상의 글을 생성하지 않는다
- 1인칭 경험 서술을 사용하지 않는다
- API 키를 로그에 출력하지 않는다

## 출력 양식
모든 블로그 글은 H1 > H2 > H3 구조를 따르며,
서론은 반드시 에피소드로 시작한다.
```

**2. 규정 문서 및 양식**
AI가 참조할 과거 데이터, 반드시 지켜야 할 출력 양식 파일을 폴더에 배치한다. 에이전트가 이 파일들을 "읽고" 작업하면, 매번 프롬프트에 설명할 필요가 없다.

**3. 스킬 파일 (SKILL.md)**
반복되는 고급 작업을 하나의 패키지로 묶은 것. 에이전트가 스킬을 "장착"하면 해당 영역의 전문가처럼 작동한다. Agent University 자체가 하나의 스킬이다.

### 실습 과제
1. 블로그 자동화 프로젝트용 CLAUDE.md를 작성한다
2. 4대 페르소나 프롬프트 파일을 규정 문서로 배치한다
3. 에이전트에게 동일한 작업을 CLAUDE.md 유/무로 각각 실행시켜 품질 차이를 비교한다

### 자가검증
- [ ] CLAUDE.md가 프로젝트 목적, 기술 스택, 금기를 포함하는가?
- [ ] CLAUDE.md 유무에 따른 출력 품질 차이를 설명할 수 있는가?
- [ ] 규정 문서가 에이전트의 작업 폴더에 올바르게 배치되었는가?

## 레슨 12: 프로젝트 분화 — 18개 부서를 운영하는 법

### 학습목표
업무 성격에 따라 독립된 프로젝트 폴더를 설계하고, 각 폴더가 자율적으로 운영되도록 구성한다.

### 핵심 원칙
모든 일을 하나의 창에서 시키면, 에이전트의 컨텍스트가 오염된다. 블로그 글을 쓰다가 주식 봇 코드를 만지면, 톤이 섞이고 실수가 난다. 업무별로 폴더를 쪼개면 각 에이전트가 "전문가"가 된다.

### 폴더 구조 예시

```
~/my-bot/
├── blog-auto/          # 블로그 자동화 (Researcher → Writer → Editor → Publisher)
├── youtube-shorts/     # 유튜브 쇼츠 파이프라인
├── stock-bot/          # 키움 API 자동매매 봇
├── ppt-generator/      # PPT 자동 생성
├── data-crawling/      # 데이터 수집 크롤러
├── welfare119/         # Welfare119 서비스
├── agent-university/   # 에이전트 교육 시스템
└── ...
```

### 각 폴더의 독립성
- 각 폴더에 자체 CLAUDE.md 배치
- 각 폴더에 자체 .env (API 키 분리)
- 각 폴더에 자체 requirements.txt
- 폴더 간 의존성 최소화

### 라우터 에이전트 (자비스)
감독자의 명령을 적절한 프로젝트 폴더로 배분하는 비서실장 에이전트.
텔레그램으로 "블로그 오늘 뭐 올렸어?"라고 물으면 blog-auto 폴더의 로그를 확인해서 보고한다.

### 자가검증
- [ ] 최소 5개 이상의 독립 프로젝트 폴더를 설계할 수 있는가?
- [ ] 각 폴더에 독립적인 CLAUDE.md가 있는가?
- [ ] 라우터 에이전트의 역할을 설명할 수 있는가?

## 레슨 13: 오토 리서치 — 스스로 개선하는 에이전트

### 학습목표
에이전트가 한 번의 결과에 만족하지 않고, 스스로 품질을 높이는 자가 개선 루프를 구축한다.

### 핵심 개념
에이전트에게 결과물을 내라고만 하면 "숙제하는 학생"이 된다.
에이전트에게 평가 기준까지 주면 "자기 검열하는 전문가"가 된다.

### 자가 개선 루프 구조

```
[결과물 생성] → [평가 기준 적용] → [미달 시 자동 수정] → [기준 통과까지 반복]
```

### 실행 방법
1. **평가 기준 제시:** "이 블로그 글이 훌륭하려면 다음 기준을 만족해야 해:
   - 서론이 에피소드로 시작하는가?
   - 1인칭/2인칭 표현이 0개인가?
   - 문장 길이 변주가 3가지 이상인가?
   - SEO 키워드가 본문에 자연스럽게 3회 이상 등장하는가?"

2. **자가 개선 명령:** "이 기준에 도달할 때까지 스스로 수정하고, 몇 번 수정했는지 보고해."

3. **품질 로그:** 수정 횟수, 수정 내용, 최종 점수를 기록한다.

### GitHub 벤치마킹
구현하고 싶은 기능을 가진 GitHub 레포지토리를 찾아서 학습에 활용한다:
1. 원하는 기능을 검색한다 (예: "blog automation python")
2. 스타 수, 최근 업데이트, README 품질로 좋은 레포를 선별한다
3. 해당 레포의 핵심 로직을 분석한다
4. 자신의 시스템에 맞게 변형 적용한다

### 자가검증
- [ ] 자가 개선 루프를 1회 이상 실행했는가?
- [ ] 품질 로그에 수정 전/후 비교가 기록되었는가?
- [ ] GitHub 레포를 1개 이상 벤치마킹했는가?

## 레슨 14: 24시간 자동화 — 잠들지 않는 시스템

### 학습목표
에이전트를 Mac Mini나 서버에서 24시간 운영하는 인프라를 구축한다.

### 인프라 옵션

| 환경 | 장점 | 단점 |
|------|------|------|
| Mac Mini (로컬) | 빠른 응답, GUI 가능, 키움 API 호환 | 정전/재부팅 시 수동 복구 |
| Vultr/AWS (클라우드) | 24시간 안정, 확장 용이 | 비용 발생, 키움 API 불가 |
| 하이브리드 | 각각의 장점 활용 | 구성 복잡 |

### 스케줄링 설정

**launchd (Mac)**
```xml
<!-- ~/Library/LaunchAgents/com.blog.daily.plist -->
<plist>
  <dict>
    <key>Label</key>
    <string>com.blog.daily</string>
    <key>ProgramArguments</key>
    <array>
      <string>/usr/bin/python3</string>
      <string>/Users/me/my-bot/blog-auto/run_all.py</string>
    </array>
    <key>StartCalendarInterval</key>
    <dict>
      <key>Hour</key><integer>9</integer>
      <key>Minute</key><integer>0</integer>
    </dict>
  </dict>
</plist>
```

**텔레그램 연동**
외부에서도 메신저로 일을 시키고 결과를 보고받는다:
- "오늘 블로그 발행 상태 보고" → 발행 로그 요약 전송
- "키워드 '기초연금' 리서치해" → 리서처 실행 → 결과 전송
- "대시보드 열어" → Streamlit 대시보드 URL 전송

### 모니터링
- Streamlit 대시보드로 발행 현황, 트래픽, 에러 로그 시각화
- 에러 발생 시 텔레그램 알림
- 주간 리포트 자동 생성

### 자가검증
- [ ] 스케줄링이 설정되어 매일 자동 실행되는가?
- [ ] 에러 발생 시 감독자에게 알림이 가는가?
- [ ] 대시보드에서 발행 현황을 확인할 수 있는가?
- [ ] 텔레그램으로 원격 명령을 내릴 수 있는가?

## Stage 3 통과 조건

- 레슨 11~14 모두 자가검증 통과
- 실제로 24시간 자동 운영되는 파이프라인 1개를 구축하여 시연
- 72시간 동안 감독자 개입 없이 정상 운영된 로그를 제출
- 감독자가 "내가 안 봐도 돌아간다"고 승인하면 졸업 시험으로 진행
